REFLECTION OF AGENTIC BUILD PROCESS: 

The agent got pretty far in implementing my spec, meeting all of the expected behaviors I outlined. For instance, it greeted the user, had a login system, selected random questions, gave hints, and noted when the user answers were correct or incorrect. It followed the proper data format of the JSON file and had a clear file structure based on what I instructed (questions, scores, users, app.py, etc.). It also correctly implemented the command-line interface, following my instructions to not “over engineer” this app using HTML, CSS, APIs, or a backend. However, on the first try, it passed 4/6 of the acceptance criteria I outlined. Initially starting the app, I manually noticed two that failed:
- The program only accepts valid inputs and reprompts the user if an invalid input is given.
- If the user wants to exit early, it will cleanly do so with an appropriate exit message. 

For example, when I “control + C” to exit early, the program crashed and aborted, giving me an error. Additionally, for the true or false question that it displayed, it handled input types incorrectly. For instance, whenever I typed “true” or “false,” it would display my input as invalid input; however it would show that the valid inputs were in fact “true or “false” leading to a never ending question cycle since everything that I inputted was deemed as invalid. This was ultimately where I had to intervene for phase 2, by prompting it with this message: 

“I tested this intial raw output you gave me, but noticed that for the true or false question, even if I put in "true" or "false" it always gave me invalid input, despite showing me that "true" or "false" are the possible valid inputs. This led to a never-ending program since everything I inputted was considered invalid." 

I think a better spec could perhaps have improved it, since I didn’t actually specify what the valid input types should be for each type of question (multiplce choice, short answer, true/false) in my spec. I could have emphasized that for MC questions, the valid inputs should be A, B, C, or D, and that for true/false, it should be the strings “true" or “false,” emphasizing case sensitivty. 

The AI review outlined clearly what was seen as a pass, fail, or warn which was helpful. However I believe the organization could have been better; for example grouping all the passes, fails, and warns together, or grouping these by my acceptance criteria and other expectations I wrote in the spec, could make it easier to review and debug these issues. The AI did flag actual problems, such as the fact that the score history was human readable. It also caught the “warn” issues for the program crashing, which was something that did occur when I initially ran the program in phase 2. One thing it didn’t catch was the exiting early error. Despite the program actually crashing when the user exits early, the AI reviewer simply tagged this as "inconsistent early exit messaging." One thing I did like was how it brought up many other possibilities that I didn’t necessarily notice or check before, like minimal username validation or rating persistence. I believe this can help with bringing up issues of secruity or improving features.  

In hindsight, I think my spec could be improved by very clearly articulating how I want the program to work. Although I added specific features and behavorial expectations, I didn't fully specify some actions -- for example, the issue with valid input types. Additionally, I think I could have organized the spec better by breaking my instructions into specific tasks, grouping related items together. 

I think the plan-delegate-review workflow is better than conversational back-and-forth workflow for large projects where many features are required and the program should span multiple files. Using an AI agent that can see all files simultaneously is more efficient than needing to copy and paste code from all files to a conversational AI chatbot. I think a conversational chatbot is better when a programmer wants more information, suggestions, or brainstorming ideas--when they don't need the AI to "act" but simply provide code. I think a conversational chatbot also just gives the user more power over the code, as changes are not directly enforced but requires the user to review and paste it in first. At the same time, I think I enjoyed this agentic build process since it felt more like what I would do in the real world working in the industry. It felt like I had a greater role in the development process compared to simply using chat-based LLMs, as I was creating a detailed description rather than simply pointing out issues and prompting a chatbot. Organizing all my thoughts and ideas into a single, clear, organized file was helpful to outline the functionality of the program overall.
